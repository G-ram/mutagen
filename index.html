Mutagen

Mutagen is a (beta) cross-platform file synchronization tool designed for remote development. It can synchronize locally or over SSH.

Source Download

Installation

Mutagen can be installed by downloading the appropriate release for your platform and adding its contents to your path.

Alternatively, Homebrew users can install Mutagen with: 


$ brew install havoc-io/mutagen/mutagen

There is no need to install Mutagen on remote systems.

Usage

Mutagen is built around the concept of a synchronization session. A session synchronizes the filesystem contents at two endpoints (alpha and beta) in a continuous, bidirectional fashion. An endpoint is simply a URL that points to either a local or remote file location. A session will monitor both endpoints for changes and automatically propagate any non-conflicting changes (including deletions!) from one endpoint to the other. Any conflicts that arise can be resolved by deleting the copy of the conflicting files on the side that should be overwritten.

The Mutagen daemon is a per-user process that runs in the background and manages sessions. Any number of sessions may be running at a given time. If a session becomes disconnected from a remote endpoint, the daemon will automatically attempt to reconnect the session in the background. Mutagen is highly resilient to connection dropouts and will pick up synchronization right where it left off.

Mutagen provides a number of subcommands that allow for the creation and management of sessions, as well as management of the daemon lifecycle. The following sections outline usage of these commands.

Additional usage information is available through the built-in help: 


$ mutagen --help

Starting the daemon

The Mutagen daemon is started using the daemon command: 


$ mutagen daemon

This command is fast and idempotent, so it can safely be added to your shell initialization script (.bashrc, etc). The daemon will automatically stop at shutdown, but you can also manually stop it using mutagen daemon -s.

Creating a synchronization session

Sessions can be created using the create command: 


$ mutagen create --ignore='**/.git' /local/path user@somewhere:~/remote/path

Endpoints can be local paths or SCP-style URLs. The order of the endpoints is not important — it only determines which will be referred to as "alpha" and which will be referred to as "beta".

If authentication is required to connect to a remote endpoint, the the create command will ask for credentials.

When creating sessions, ignore specifications can be provided to ignore content within a synchronization root. When a path is ignored, Mutagen won't propagate it or modify it.

Listing sessions

The list command shows the current status of each session, as well as any conflicts or problems that have arisen in the propagation of changes between endpoints: 


$ mutagen list
Session: 7bb4a56b-f4be-4083-be24-7f590beda02e
Ignored paths:
        **/.git
Status: Watching for changes
Alpha:
        URL: /local/path
        Status: Connected
Beta:
        URL: user@somewhere:~/remote/path
        Status: Connected

Monitoring a session

The monitor command shows live monitoring information for a session: 


$ mutagen monitor 7bb4a56b-f4be-4083-be24-7f590beda02e
Session: 7bb4a56b-f4be-4083-be24-7f590beda02e
Ignored paths:
        **/.git
Alpha: /local/path
Beta: user@somewhere:~/remote/path
Staging files on alpha: 75% (8942/11782)

Pausing a session

Synchronization can be temporarily halted using the pause command: 


$ mutagen pause 7bb4a56b-f4be-4083-be24-7f590beda02e

Resuming a session

Sessions can be resumed via the resume command: 


$ mutagen resume 7bb4a56b-f4be-4083-be24-7f590beda02e

This command will ensure that a session is unpaused and attempting to synchronize. If the daemon can't automatically reconnect to an endpoint because authentication is required, the resume command can be used to provide credentials.

Terminating a session

Synchronization can be permanently halted for a session (and the session deleted) using the terminate command: 


$ mutagen terminate 7bb4a56b-f4be-4083-be24-7f590beda02e

This will not delete files on either endpoint (but should be done before completely deleting files on either endpoint to avoid propagating the complete deletion).
